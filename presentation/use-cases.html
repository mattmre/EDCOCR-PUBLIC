EDX eDiscovery & Legal Hold Scenario 01

Outside counsel collects a 50 GB document production. Every page must be Bates-stamped, every redaction must be auditable, and every extraction must survive opposing counsel's challenge in deposition.

The problem

Cloud OCR APIs produce text that may include hallucinated characters with no way to verify which model emitted them. Tesseract alone struggles on noisy scans and offers no chain-of-custody primitives. Your bates-numbered production needs to be defensible for years.

EDCOCR's answer

  • CTC-only recognition means extracted text is bounded by what was actually in the image — no fabrication.
  • Bates stamping is built in (stamps/ module) with stamp-aware custody events (BATES_STAMP_APPLIED).
  • Privilege-keyword detection runs as part of structured extraction — flagged pages route to the human review queue.
  • The custody log gives every reviewer a hash-verifiable answer to "did this text change after extraction?"
  • Air-gapped deployment supports protective orders that prohibit cloud uploads.

Operational outcome

A 50 GB production processes in hours on a single multi-GPU host (T3) or scales to a multi-tenant SaaS bureau on Kubernetes (T6). Output flows directly into Relativity, Everlaw, or your review platform of choice via the searchable PDF + sidecar JSON.

LEG Law Firm Internal Operations Scenario 02

A mid-sized firm needs to digitize legacy paper files, make all contracts text-searchable, and extract key fields (effective date, parties, term) for the matter-management system. The data cannot leave the firm's network.

The problem

Contract review tools that demand cloud upload run into client confidentiality issues. On-prem OCR options that don't ship structured-extraction integrations leave the firm with text but no fields. The firm wants both, on infrastructure they control.

EDCOCR's answer

  • Self-hosted, Apache 2.0 — no per-page billing, no cloud upload, no SaaS subscription.
  • Structured extraction via PaddleNLP UIE (or regex fallback) emits dates, amounts, names, addresses to EXPORT/EXTRACTION/.
  • Document classification routes contracts vs litigation correspondence vs general business records to the right extraction profile.
  • Webhook delivery pushes results into the matter-management system as each document finishes.

Operational outcome

A single mid-range GPU server (T1) backed by a NAS for the document store covers a 50-attorney firm comfortably. The Python SDK gives the IT team a clean handle to script integrations against.

FOR Digital Forensics & Investigations Scenario 03

A forensic examiner is processing a seized drive containing scanned ledgers, photographed documents, and PDFs of varying quality. Every page must be preserved exactly as found, and every operation must be reproducible at trial.

The problem

Cloud OCR APIs drop pages they can't read. They mutate the source. They sit behind terms of service that prohibit forensic use. Their model versions are opaque, so you cannot say in court which model emitted which character on which day.

EDCOCR's answer

  • Forensic image preservation: pages that fail OCR are embedded as images in the output PDF — evidence is never discarded.
  • Hash-chained custody log captures the exact PaddleOCR model version, DPI, engine fallback chain, and confidence per page.
  • Deterministic processing: same input + model + config = byte-identical output (modulo timestamps).
  • Air-gapped deployment supports SCIF environments and chain-of-custody requirements that forbid network exposure.
  • Handwriting detection and signature detection (advisory) flag manual-review items separately.

Operational outcome

A single GPU workstation (T1) or air-gapped bundle (T7) gives the examiner a portable, defensible toolkit. The custody log can be exported to the case-management system and verified by hash from any reviewer.

GOV Government & FOIA Backlog Scenario 04

A federal agency has a multi-year FOIA backlog of scanned correspondence, handwritten memos, and multi-language records. Generative AI is contractually off the table. The pipeline must be auditable end-to-end.

The problem

Procurement constraints rule out cloud OCR APIs. Internal IT does not have the headcount to build OCR from scratch. The volume — millions of pages — needs to scale horizontally. The languages — English, Spanish, Arabic, Chinese — need real coverage, not best-effort.

EDCOCR's answer

  • Apache 2.0 license; no per-page billing; can be deployed inside the agency's existing Kubernetes footprint.
  • 34 core languages with FastText-driven model swapping handles mixed-language documents out of the box.
  • Helm chart with KEDA autoscaling lets the backlog burst-process during off-hours.
  • Handwriting detection flags handwritten memos for the human review queue separately from clean-print correspondence.
  • No outbound network calls at runtime — fully compatible with FedRAMP-class network posture.

Operational outcome

A Kubernetes deployment (T6) can churn through millions of pages over weeks with full audit trail. The agency's existing Prometheus / Grafana stack picks up EDCOCR's metrics natively.

HLT Healthcare Records Digitization Scenario 05

A hospital system needs to digitize legacy patient charts, extract PHI for the EHR migration, and produce an auditable trail that satisfies the compliance officer. The data cannot touch a public cloud at any point.

The problem

HIPAA-class restrictions take cloud OCR off the table. Existing on-prem tools may not extract spatial PHI (bounding boxes on the original page) with the fidelity needed for downstream redaction. Per-tenant isolation matters when multiple departments share infrastructure.

EDCOCR's answer

  • PII/PHI spatial extraction emits bounding boxes on the original page, so downstream redaction tools know exactly what to mask.
  • Per-tenant isolation enforced at API, DB, and output-path level — no leakage between departments.
  • Air-gap deployment with pre-baked models supports HIPAA-class network postures.
  • Custody log gives the compliance officer a hash-verifiable trail.
  • Output retention policy (configurable) integrates with the records-management retention schedule.

Operational outcome

A distributed deployment (T4 or T6) on the hospital's on-prem cluster covers a multi-facility system. PHI extraction feeds the EHR migration tool with bounding boxes for accurate masking.

INS Insurance Claims & Underwriting Scenario 06

A carrier needs to process claim packets — photographs of damaged property, scanned medical bills, handwritten driver statements, multi-page forms. Field extraction must be accurate enough to flag fraud signals automatically.

The problem

Claim packets are heterogenous — photos, forms, scans, handwriting all in one bundle. Cloud OCR APIs charge per page and have to be carefully audited for compliance. Internal teams want a self-hosted pipeline that produces consistent structured fields.

EDCOCR's answer

  • 18 image formats + PDF support handle the heterogeneous packet without preprocessing.
  • Document classification routes forms vs photographs vs medical bills to the right downstream extractor.
  • Structured extraction emits dates, amounts, names, addresses — the fields the claims system needs.
  • Handwriting detection flags handwritten statements for human review separately from machine-print pages.
  • Exception routing rules engine fires on low confidence, mismatched values, or fraud-signal keywords.

Operational outcome

A T4 distributed deployment handles a national carrier's intake stream. Webhook delivery pushes results into the claims platform in real time. The review queue handles edge cases without bottlenecking the main pipeline.

REG Regulated Document Archive Scenario 07

A financial services firm archives signed contracts, board minutes, and regulatory filings. Records must be text-searchable for the audit team and retrievable on demand for the regulator. Retention rules apply.

The problem

Archives that are stored as image PDFs are not searchable. Re-OCR'ing them with a cloud tool re-uploads sensitive documents. The audit team needs a unified retrieval surface so they can answer regulator questions in hours, not days.

EDCOCR's answer

  • Re-processes archived PDFs in place, adding a searchable text layer and metadata sidecars.
  • Unified retrieval output (EXPORT/RETRIEVAL/) emits a structured JSON + Markdown package optimized for vector search or full-text indexing.
  • Entity recall API surfaces all instances of an entity across the archive (e.g., every document mentioning a counterparty).
  • Custody log gives the auditor a defensible answer to "when was this document indexed, with what model?"
  • Per-tenant cost tracking gives the CFO a per-department billing surface.

Operational outcome

A multi-year archive of millions of pages processes through a Kubernetes deployment (T6) over a controlled rollout window. Output integrates with the firm's vector search provider, RAG pipeline, or full-text index of choice.

Where EDCOCR is not a fit

Honest scoping. Pick a different tool for these.

Pure document understanding without provenance needs

  • If you need pure document-understanding chat ("summarize this") and don't care where the text came from, a generative VLM like GPT-4V or Claude with vision will be faster and more flexible.
  • EDCOCR optionally integrates with VLMs (Phase 4 VLM gateway), but the core is not built around that workflow.

Real-time consumer document scanning

  • If you need a mobile app to OCR a receipt in 200ms with no server, a small on-device model is the right choice.
  • EDCOCR is built for batch processing where each page may take 0.5–2 seconds and produces 14 output families with full custody.

Form auto-fill against a fixed template

  • If you have a single form to OCR (one PDF, fixed coordinates), specialized form-extraction tools like the AWS Textract Forms API can be configured faster.
  • EDCOCR is the right choice when the documents are heterogenous, the fields are not at fixed positions, or the document set spans languages and domains.
