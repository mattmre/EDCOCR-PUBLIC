01Overview

EDCOCR is a GPU-accelerated OCR platform built around a six-stage producer-consumer pipeline. The runtime never makes outbound network calls — all models ship pre-baked into the Docker image. The output of every job is byte-deterministic given the same input, model versions, and configuration.

The platform is structured as three concentric layers:

  1. Forensic core — Deterministic OCR (PaddleOCR CTC), image preservation, hash-chained custody, language detection. Boundary-validated against AI substitution.
  2. AI-adjacent — NER, classification, structured extraction, translation. Feature-gated, custody-aware, and explicitly separated from the forensic core.
  3. Operational — REST API, WebSocket/SSE streaming, Prometheus metrics, Grafana dashboards, distributed coordinator, KEDA autoscaling.
Boundary contract: A drift-guard script (scripts/validate_feature_boundary.py) enforces the forensic-core / AI-adjacent separation in CI. Generative models cannot be substituted into the OCR loop without tripping the validator.

02Pipeline architecture

The monolithic pipeline (ocr_gpu_async.py) runs six stages concurrently across 31 threads. Bounded queues sit between every stage; when a downstream stage falls behind, upstream stages block rather than dropping work.

Stage 1

Scheduler

1 thread

Scans ocr_source/, creates chunk tasks, handles page-level resume from temp directory.

Stage 2

CPU Extractors

8 threads

PDF/image to PIL images at 300 DPI. Adaptive DPI escalation on low confidence.

Stage 3

GPU Workers

12 threads

PaddleOCR (CTC) → Tesseract → image-only fallback. Optional PP-StructureV3 layout.

Stage 4

Assembler

1 thread

Collects pages via document registry, merges into final PDF, emits sidecars.

Stage 5

Compressors

8 threads

Ghostscript optimization with integrity validation. 300s subprocess timeout.

Stage 6

Monitor

1 daemon

Real-time PPM, docs/hour, registry size. Docker healthcheck heartbeat.

Why threading, not asyncio?

The workload is CPU-bound (image decoding), GPU-bound (OCR inference), and subprocess-bound (Ghostscript). asyncio's main advantage — cheap context switching across many I/O-waits — doesn't help here. Threading with bounded queues gives explicit backpressure and easier reasoning about shutdown.

Crash resume

Per-page PDFs land in ocr_temp/<doc_hash>/ as they complete. A SIGTERM mid-batch leaves the temp directory intact; the next run picks up at the first missing page. compute_resume_gap_chunks() performs the set-difference deterministically.

03Tech stack

OCR engine
PaddleOCR 2.9.1 with CTC decoding. Per-language model swap driven by FastText detection. Fallback chain: PaddleOCR → Tesseract → image-only. Optional use_onnx=True for CPU acceleration (4–7×).
PDF I/O
PyMuPDF (fitz) for read/write, pdf2image + Poppler for rasterization, Ghostscript for compression with integrity verification.
Language detection
FastText with lid.176.bin model. Two-pass: pre-OCR sample for engine selection, post-OCR full text for sidecar emission. Per-span detection optional.
API server
FastAPI 0.135+ + Uvicorn + slowapi for rate limiting + SQLAlchemy 2.x for the job DB. WebSocket + SSE both supported.
Coordinator
Django 5.2 + Celery 5.6 + RabbitMQ (quorum queues) + Redis (cache + Streams + Sentinel) + PostgreSQL (job state).
NLP / Extraction
spaCy 3.8 for NER (with custom forensic types), PaddleNLP UIE for structured extraction, regex fallback for offline / air-gapped operation.
Layout intelligence
PP-StructureV3 for layout + tables. Optional LayoutLMv3 fine-tuned adapter for domain-specific document understanding.
Container runtime
Multi-stage Dockerfile with non-root user (UID 1000), capabilities dropped, seccomp default. CUDA 12 base for GPU workers; python:3.11-slim for CPU workers.
Orchestration
Docker Compose (single host), Helm chart with 26 templates (Kubernetes), Terraform modules for EKS / GKE / OKE.
Observability
Prometheus custom collector (7 metric families) + Grafana 53-panel dashboard + 5 alert rules. OpenTelemetry traces. Hash-chained JSONL audit log.
SDKs
edcocr-sdk (Python 3.10+) and @edcocr/sdk (TypeScript, zero deps). Versioned in lockstep with the server.

04Language handling

45 languages across two tiers, controlled by OCR_LANGUAGE_TIERS=core[,extended]:

Core tier — 34 languages (default)

  • Latin: en, fr, de, es, it, pt, nl, sv, da, fi, ro, pl, cs, hu, tr, vi
  • Cyrillic: ru, uk, be, bg
  • CJK: ch (Simplified), chinese_cht (Traditional), japan, korean
  • Arabic / RTL: ar, fa, ur, ug
  • Indic: hi, ta, te, kn
  • Other: el (Greek), ka (Georgian)

Extended tier — +11 languages (opt-in)

  • Latin: hr, sk, no, lt, lv, et, rs_latin
  • Indic: bn (Bengali), mr (Marathi), ne (Nepali)
  • Southeast Asian: th (Thai)

Per-span detection

Opt-in via ENABLE_PER_SPAN_LANGUAGE=true. Emits EXPORT/LANGUAGE/<doc>.language.json with BCP-47 tags at line, page, and document levels. Re-detect endpoint (POST /api/v1/jobs/{id}/redetect-language) reruns detection on the embedded text layer without re-running OCR.

06Custody log

Every operation emits a custody event to a hash-chained JSONL log. Each event records the operation type, ISO 8601 timestamp, job ID, page index, tenant ID, operation-specific payload, and the SHA-256 hash of the previous event.

07Distributed mode

For horizontal scale, EDCOCR runs as a Django coordinator plus a Celery worker fleet. Workers register their capabilities on startup; the coordinator routes tasks based on registered capability. KEDA scales worker replica counts based on queue depth.

Multi-GPU affinity

With ENABLE_PER_GPU_QUEUES=true and GPU_COUNT=N, the router creates round-robin ocr_gpu_{0..N-1} queues so workers pin to specific GPUs with no cross-card thrash.

Storage backends

Workers can read source files via direct path (local), shared mount (NFS), or presigned URL (S3 / MinIO). In S3 mode, workers never see AWS credentials — the coordinator generates presigned URLs per task.

08Deployment topologies

Seven topology IDs, each documented with its own decision criteria. See DEPLOYMENT-DECISION-GUIDE.md for the full tree.

T1

Single GPU host

  • 1 GPU, 16+ GB VRAM
  • Docker Compose
  • ≤ 5k pages/day
  • Lab / pilot / small team
T2

Single CPU host

  • No GPU, ONNX backend
  • Docker Compose
  • ≤ 2k pages/day
  • Budget-constrained pilots
T3

Multi-GPU host

  • 2–8 GPUs, per-GPU queues
  • Docker Compose
  • ≤ 50k pages/day
  • Single-server power-user
T4

Distributed Celery

  • Coordinator + N workers
  • NFS or S3 storage
  • ≤ 500k pages/day
  • Department / mid-market
T6

Kubernetes

  • Helm chart (26 templates)
  • KEDA autoscaling + HA
  • 500k+ pages/day
  • Enterprise / SaaS
T7

Air-gapped

  • Pre-baked bundle
  • No outbound calls
  • SCIF / classified
  • airgap-bundle.sh + airgap-deploy.sh

Throughput levers

  • NUM_WORKERS — match to VRAM; oversubscription causes OOM
  • Adaptive batching — complexity-aware dynamic batch sizing
  • Page cache — memory-mapped LRU for repeated reads
  • Engine selection — quality-based routing to Tesseract for clean print (faster) and Paddle for complex layouts

Python SDK

edcocr-sdk 
from edcocr_sdk import EDCOCRClient

c = EDCOCRClient(base_url="https://ocr.example.com", api_key="")
job = c.submit_job(path="contract.pdf")
for ev in c.watch(job.id):
    print(ev.progress_pct, ev.stage)
result = c.fetch_outputs(job.id)

TypeScript SDK

@edcocr/sdk 
import { EDCOCRClient } from "@edcocr/sdk";

const c = new EDCOCRClient({
  baseUrl: "https://ocr.example.com",
  apiKey: process.env.EDCOCR_API_KEY!,
});
const job = await c.submitJob({ file: "contract.pdf" });
for await (const ev of c.watch(job.id)) {
  console.log(ev.progress_pct, ev.stage);
}
const outputs = await c.fetchOutputs(job.id);
