1. System Overview

EDCOCR is a producer-consumer document processing platform with a single production entry point (ocr_gpu_async.py) and three deployment shapes. The platform is structured as a strict core forensic plane (CTC OCR, custody log, output assembly) surrounded by AI-adjacent extensions (NER, classification, structured extraction, translation). The boundary is enforced by scripts/validate_feature_boundary.py.

The split matters because forensic workflows can accept the CTC OCR result as evidentiary, but the AI-adjacent outputs (NER, classification, etc.) are always advisory and require human review before use as evidence.

2. Production Pipeline (6 Stages)

The async pipeline runs six stages concurrently, with bounded queues between them providing backpressure. All work is doc/page-granular so a crash never loses more than one page of progress.

maxsize=200| GW[GPU Workers x12] GW -->|page_queue
unbounded| AS[Assembler] AS -->|compression_queue
unbounded| CP[Compressors x8] CP --> OUT[Final PDFs] MON[Monitor daemon] -.heartbeat.-> HC[Healthcheck] GW -.metrics.-> MON AS -.metrics.-> MON
Stage Threads Responsibility Output
1. Scheduler 1 Scan source, hash path -> doc_id, create chunk tasks, page-level resume Chunk tasks
2. Extractors 8 PDF/image -> PIL @ 300 DPI, image preprocessing if degraded PIL image batches
3. GPU Workers 12 PaddleOCR detection + recognition, language model swap, Tesseract fallback, NER + classify + extract Page result + custody event
4. Assembler 1 doc_registry merge, finalize PDF, emit sidecar JSON (10+ schemas) Final PDF + sidecars
5. Compressors 8 Ghostscript optimization, integrity validation, atomic rename Deliverable PDFs
6. Monitor 1 daemon Heartbeat, PPM metrics, healthcheck file write Liveness signal

Defaults assume a single 16-24GB GPU. Scale NUM_WORKERS based on VRAM; the other knobs follow throughput tuning.

3. Coordinator Data Model

In distributed mode, the Django coordinator persists job state to PostgreSQL. The schema is intentionally narrow: enough to coordinate workers and answer "where is my document?" without becoming an evidence store.

Core Entity Relationships
erDiagram TENANT ||--o{ JOB : owns JOB ||--o{ PAGE_RESULT : has JOB ||--o{ CUSTODY_EVENT : has JOB ||--o{ PII_ENTITY : has WORKER ||--o{ PAGE_RESULT : produced WORKER { string worker_id PK string capability "ocr_gpu / ocr_cpu / nlp / compress" string status "online / offline / draining" datetime last_heartbeat } JOB { uuid id PK string tenant_id FK string status "pending / processing / done / failed" int priority "0..9" datetime created_at json settings } PAGE_RESULT { uuid id PK uuid job_id FK int page_number float confidence string quality "high / medium / low" string engine "paddleocr / tesseract / image-only" } CUSTODY_EVENT { uuid id PK uuid job_id FK string event_type string prev_hash "SHA-256" string this_hash "SHA-256" datetime timestamp json payload }

Notably absent: an OCR text column. Extracted text lives in the output filesystem (NFS or S3), not in the coordinator DB. Keeps the DB small, the deliverable canonical, and recovery from a failed coordinator simple.

4. Deployment Topologies

Three deployment patterns, same container image, same OCR logic.

Single Machine

One container, GPU optional, watches a folder. Fits on a workstation.

Best for: pilot, dev, small teams, < 10k pages/day.

Distributed (K8s)

Coordinator + N workers, KEDA-driven autoscaling on queue depth.

Best for: production, multi-tenant, > 10k pages/day.

Air-Gapped

Bundled artifact, no outbound traffic, models pre-baked.

Best for: SCIF, classified, regulated environments.

6. REST & Realtime Surface

FastAPI server with API-key auth, rate limiting, request size limits, and three real-time channels: WebSocket per job, SSE for fleet events, and HMAC-signed webhook delivery.

API Request Lifecycle
sequenceDiagram participant C as Client participant A as Auth Middleware participant R as Rate Limiter participant J as Job Manager participant Q as Celery Queue participant W as Worker participant WH as Webhook Engine C->>A: POST /api/v1/jobs A->>A: Verify X-API-Key A->>R: Check rate limit R->>J: Submit job J->>J: Validate, persist J->>Q: Enqueue J-->>C: 201 + job_id Q->>W: Dispatch W->>W: Process W->>J: Update status J->>WH: Trigger webhook WH->>C: HMAC-signed POST

Endpoint families: jobs (CRUD + retry), outputs (manifest + download), recall (entity / extraction search), review (human review queue), health (liveness + detailed), metrics (Prometheus). All endpoints are tenant-scoped.

9. Chain of Custody

The custody log is a SHA-256 hash-chained JSONL file. Each event records prev_hash, hashed inputs, and a this_hash. The log is append-only; verification is a linear scan that recomputes hashes.

Event Emission Points
sequenceDiagram participant FS as Filesystem participant SC as Scheduler participant GW as GPU Worker participant ASM as Assembler participant CL as Custody Log FS->>SC: New file SC->>CL: FILE_INGESTED (SHA-256 of source) SC->>GW: Chunk task GW->>CL: PAGE_OCR_OK or PAGE_OCR_FAILED GW->>CL: LANGUAGE_DETECTED (per page) GW->>CL: NER_EXTRACTED (entity counts) ASM->>CL: DOCUMENT_FINALIZED (output SHA-256) ASM->>CL: COMPRESSION_COMPLETE

Event taxonomy: FILE_INGESTED, PAGE_OCR_OK, PAGE_OCR_FAILED, LANGUAGE_DETECTED, LANGUAGE_MIXED_SCRIPT, LANGUAGE_REDETECTED, NER_EXTRACTED, CLASSIFIED, HANDWRITING_DETECTED, SIGNATURE_DETECTED, DOCUMENT_FINALIZED, TRANSLATION_REJECTED, CERTIFICATION_GRANTED, CONFIG_APPLIED, and twenty-plus more in the full schema.

10. Failure & Resume Modes

FailureBehaviorRecovery
Process crash mid-document Per-page PDFs in ocr_temp/<doc_hash>/ Restart resumes from last completed page
OCR fails on a page Adaptive DPI escalation -> 450 DPI -> 600 DPI If all retries fail, embed original image; custody event recorded
Tesseract / Ghostscript hang Subprocess timeout (120s / 300s) kills the process Page enters failure pipeline; document continues
GPU OOM Adaptive batch sizing reduces batch; retry Falls back to single-page if needed
Coordinator DB down Celery retries with exponential backoff Workers continue from queued tasks once DB recovers
RabbitMQ failover Quorum queues hold messages across broker restarts Auto-reconnect; no message loss
Worker pod killed Page-level resume + ack-late semantics Another worker picks up unacked tasks

The design principle: never silently lose evidence. Failure preserves the original image, records a custody event, and lets the operator decide whether to retry, escalate, or accept the failed page as-is.

11. Translation Suite (Opt-In)

Pluggable engine adapters behind a policy router. All translation flags default OFF; enabling requires explicit per-tenant policy and a 48-hour bake before any flag-flip PR is considered for production.

Engine Selection Flow
flowchart TB REQ[Translation Request] --> POL[TenantPolicy] POL -->|allow_nc_licensed=false| RANK[Rank Candidates] POL -->|allow_nc_licensed=true| RANK RANK --> FILTER[Filter NC-licensed if needed] FILTER --> ROUTE{Route?} ROUTE -->|privileged content| LOCAL[Local engines only] ROUTE -->|normal content| ANY[Any engine] LOCAL --> OPUS[OPUS-MT] LOCAL --> NLLB[NLLB-200] LOCAL --> MADLAD[MADLAD-400] ANY --> OPUS ANY --> NLLB ANY --> MADLAD ANY --> LLM[LLM Tier] OPUS --> CUSTODY[Custody Event] NLLB --> CUSTODY MADLAD --> CUSTODY LLM --> CUSTODY

Privilege-blocked content (legal documents matching configured keywords) never routes to cloud generative engines. The decision is logged before the engine call so the audit trail records intent, not just outcome.
